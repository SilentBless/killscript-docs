# Работа с документацией

## Главное правило

В публичную документацию попадает только API, проверенный непосредственно в игре. Старая русская документация помогает составить перечень проверок, но не считается источником истины.

## Перед публикацией API

- подтвердить имя и тип;
- отдельно проверить чтение и запись свойства;
- подтвердить Client, Server, Reflex и другие контексты;
- проверить аргументы и возвращаемое значение;
- зафиксировать фактическое поведение и ограничения;
- запустить полный публикуемый пример;
- написать русскую страницу;
- корректно перевести готовую страницу на английский.

Если логический раздел проверен не полностью, он остаётся в рабочем списке и не публикуется.

## Структура API-страницы

1. Краткое назначение.
2. Доступность по контекстам.
3. Быстрый пример.
4. Свойства с типом и доступом `get`/`set`.
5. Методы и сигнатуры.
6. Фактическое поведение и ограничения.
7. Типичные ошибки.
8. Полный проверенный пример.
9. Связанные API.
10. Версия игры, на которой выполнена проверка.

## Структура учебной страницы

1. Ожидаемый результат.
2. Необходимые знания.
3. Структура файлов.
4. Пошаговая реализация.
5. Полный код.
6. Объяснение важных частей.
7. Проверка результата.
8. Частые ошибки.
9. Следующие шаги.

Учебные материалы и API Reference не смешиваются: первые учат, второй помогает быстро найти точную техническую информацию.
